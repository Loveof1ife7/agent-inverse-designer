# Closed-Loop Discovery Context

This document defines the current closed-loop design. README updates, interface docs, demo descriptions, and code migration should follow this document.

Core change:

```text
old:
  AgentExplorer designs datagen meta/config
  DatagenFEMEvaluator generates online candidate structures

new:
  AgentExplorer designs a target schedule
  InverseDesigner reads targets and generates explicit structures
  FEMEvaluator validates the generated structures
```

`DatagenFEMEvaluator` is no longer part of online closed-loop exploration. Its main role is offline cold start: generate large pretraining data, for example 500k structure-property samples, to pretrain `InverseDesigner`.

## 1. Division of Labor

```text
DatagenFEMEvaluator:
  offline data factory
  generate large pretraining dataset
  not part of online closed loop

InverseDesigner:
  target -> explicit structure
  explores target space by sampling structures for scheduled targets
  updates itself from new FEM-validated samples

AgentExplorer:
  reads Knowledge + FeedbackSignal
  proposes target schedule S_n = [T_1, T_2, ..., T_k]
  does not directly tune datagen parameters in the closed loop

FEMEvaluator:
  explicit structure -> true evaluated property / validity / raw metrics
  uses Abaqus when fem_backend=abaqus, or fem_backend=auto with Abaqus available
  uses proxy only for fast tests, demos, or auto fallback when Abaqus is unavailable

RawExperimentStore:
  append-only record of what happened

KnowledgeBase:
  target-space exploration memory
  stores schedule evidence, inverse-designer capability, FEM outcomes, and provenance

FeedbackSignal:
  short-term control summary for the current final target
```

## 2. Offline Cold Start

```text
DatagenFEMEvaluator
  -> generate large structure dataset
  -> FEM evaluate, with proxy allowed only as a cold-start/testing fallback
  -> build pretraining dataset
  -> pretrain InverseDesigner
  -> optionally seed KnowledgeBase
```

This stage gives `InverseDesigner` a broad initial mapping:

```text
property / target -> explicit structure
```

After this stage, the online loop should not call datagen to create new search candidates.

## 3. Online Closed Loop

Input:

```text
final target T
```

Loop:

```text
1. AgentExplorer reads:
     - final target T
     - FeedbackSignal from the previous round
     - KnowledgeBase / refined knowledge

2. AgentExplorer proposes target schedule:
     S_n = [T_1, T_2, ..., T_k]

3. InverseDesigner samples structures:
     T_i -> structure_i

4. FEMEvaluator evaluates structures:
     structure_i -> evaluated_property_i, validity_i, raw_metrics_i
     explicit coordinate/edge structures are materialized as truss txt before Abaqus evaluation

5. RawExperimentStore writes observations:
     final_target
     scheduled_target
     schedule_meta
     structure
     evaluated_property
     validity
     provenance

6. Knowledge projector writes KnowledgeEvidence:
     schedule -> structure -> FEM property -> errors -> label

7. Dataset curator updates InverseDesigner replay / finetune data.

8. FeedbackSignalExtractor compresses this round into scheduler control:
     success_found / should_stop
     best_success / best_near_miss / best_sample
     next_anchor
     main_error_direction
     failure_modes
     suggested_next_action

9. Stop if success, otherwise continue with a new schedule.
```

## 4. Target Schedule

A schedule is the AgentExplorer action in the new loop.

```text
TargetSchedule:
  schedule_id
  final_target
  scheduled_targets
  strategy
  hypothesis
  reason
  confidence
  samples_per_target
```

Typical strategies:

```text
exploitation:
  sample near final target T

reachability_probe:
  test targets that may be easier for InverseDesigner

boundary_probe:
  explore high/low property regions

counterfactual:
  test a target offset that should fail if the current hypothesis is wrong

curriculum:
  move from reachable targets toward final target T
```

## 5. Observation Semantics

Each evaluated structure should preserve both target levels:

```text
final_target:
  the user objective T

scheduled_target:
  the target T_i chosen by AgentExplorer for this sample

evaluated_property:
  FEM result of the structure generated by InverseDesigner
```

Two errors are required:

```text
error_to_scheduled_target:
  how well InverseDesigner satisfied the requested T_i

error_to_final_target:
  how useful this sample is for solving final target T
```

This distinction is central. It lets the system learn both:

```text
1. InverseDesigner capability map:
   where it can generate reliable structures

2. Schedule utility map:
   which target-space moves help reach final target T
```

## 6. Knowledge Role

Knowledge is not a generic log. It is a target-space exploration map.

It answers:

```text
Which scheduled targets are reachable?
Where does InverseDesigner show systematic bias?
Which target offsets move FEM properties closer to final target T?
Which schedule strategies repeatedly fail?
Which observations should be used for finetuning?
```

Knowledge has three jobs:

```text
1. guide AgentExplorer target schedule design
2. model InverseDesigner capability boundaries
3. curate finetune / replay data for InverseDesigner
```

## 7. Dataset vs Knowledge

```text
Dataset:
  training material for InverseDesigner
  target/property -> explicit structure
  weighted by validity, novelty, error, and coverage

Knowledge:
  decision memory for AgentExplorer and scheduler
  final_target + scheduled_target + structure + FEM property + evidence
```

They can come from the same raw observations, but they are different projections.

## 8. FeedbackSignal vs Knowledge

```text
FeedbackSignal:
  current round
  current final target T
  short-term controller state
  stop / continue / next anchor / failure summary

Knowledge:
  all rounds
  long-term memory
  schedule evidence and target-space patterns
```

FeedbackSignal tells the loop what to do next. Knowledge tells AgentExplorer what has been learned.

FeedbackSignal is local and deterministic. It only summarizes the current round against final target `T`; it does not replace KnowledgeBase, KnowledgeRefiner, or AgentExplorer reasoning.

## 9. Implementation Direction

New or renamed contracts should move toward:

```text
TargetSchedule
TargetScheduleCandidate
TargetScheduleProposal
TargetScheduleObservation
```

Legacy contracts such as `DatagenConfig` and `MetaCandidate` may remain during migration, but their closed-loop role is legacy.

The final scheduler should call:

```text
AgentExplorer.propose_target_schedule(...)
InverseDesigner.sample_schedule(...)
FEMEvaluator.evaluate_explicit_structures(...)
RawExperimentStore.append_many(...)
KnowledgeBase.add_knowledge_evidences(...)
InverseDesigner.update_replay_or_finetune_dataset(...)
```

## 10. One Sentence

```text
AgentExplorer learns how to choose targets; InverseDesigner learns how to realize targets; FEM supplies truth; Knowledge records which target-space moves work.
```
